# DA Orchestrator (공공DB 데이터아키텍트 파이프라인)

공공데이터베이스 DA의 7단계를 순차 위임으로 조율하여 표준 준수 데이터구조와 산출물을 생성하는 통합 스킬. 판단 주체·행동 원칙은 `da-agent` 에이전트가, 절차·조율·게이트는 이 스킬이 정의한다.

## 실행 모드: 서브 위임 (순차 파이프라인)

- 각 단계를 `Skill`(스킬) 또는 `Agent`/`SendMessage`(metadata-agent)로 호출하고 산출물(파일)로 결과를 수집한다.
- 팀(`TeamCreate`) 모드가 아니다 — 단계가 이전 결과에 의존하므로 **순차 실행 + 환류**한다.
- **Agent 호출 시 model 명시**: `metadata-agent`는 `Agent(subagent_type: "metadata-agent", model: "sonnet")`로 호출한다(표준 판정은 sonnet으로 충분, 비용 효율 — agent-design-patterns의 opus 권고에 대한 의도적 절충). 스킬 단계는 `Skill` 도구로 호출한다.

## 실시간 모니터링 (진행 가시화·게이트)

오케스트레이터는 위임 워크플로우의 진행 상태를 **TodoWrite + 산출물 파일**로 실시간 추적한다.

- **TodoWrite로 7단계 등록**: Phase 1 진입 시 7개 단계(데이터모델링→표준화검토→데이터구조검증→산출물관리→변경관리→품질관리→데이터분석)를 TodoList로 등록한다. 부분 재실행 시에는 재실행 대상 단계와 하위 의존 단계만 등록한다.
- **단계 경계마다 갱신**: 단계 진입 시 해당 항목을 `in_progress`, 산출물 생성·게이트 통과 시 `completed`로 즉시 갱신한다. 동시에 `in_progress`는 1개만 유지한다.
- **게이트 환류 시 되돌림**: 3·6 게이트에서 `보완/부적합/NCR`로 환류하면, 환류 대상 단계(1·2 등)를 다시 `in_progress`로 되돌리고 재실행 결과를 반영한다.
- **산출물 파일 = 완료 신호**: 각 단계 완료는 작업공간의 산출물 파일 존재·내용(Read/Glob)으로 검증한다. TodoWrite `completed` 표시 전에 해당 산출물 존재를 확인한다(표시만 하고 파일 없는 상태 금지).
- **metadata-agent 환류는 SendMessage**: 2단계 `확인 필요` 발생 시 동일 에이전트에 `SendMessage`로 재질의하여 컨텍스트를 유지한다(새 `Agent` 호출로 재시작하지 않는다).

## 사용 시점

- `{요구사항번호}-db-modeling.md` 입력 또는 요구사항으로 DB 산출물 일괄 생성을 요청받을 때
- "데이터 모델링/DB 설계/데이터 산출물 생성/ERD·정의서"를 요청받을 때
- **후속**: 재실행, 부분 수정, 모델 보완, 표준검토 반영, 산출물 갱신 등 기존 결과 개선 요청 시

## 에이전트/스킬 구성

| 단계 | 호출 | subagent_type/skill | 입력 | 출력 |
|---|---|---|---|---|
| 1. 데이터모델링 | Skill | `data-modeling` | 요구사항·대상DBMS | `…-db-modeling.md` |
| 2. 표준화검토 | Agent | `metadata-agent` | 1의 산출물 | `…-표준검토결과.xlsx` |
| 3. 데이터구조검증 | Skill | `data-structure-verification` | 2 반영 모델·DBMS | `…-구조검증결과.md` |
| 4. 산출물관리 | Skill | `data-output-management` | 1·2·3 결과 | **산출물 7종 전부**(논리/물리ERD·엔터티·애트리뷰트·DB·테이블·컬럼정의서) + `…-산출물관리대장.xlsx` |
| 5. 변경관리 | Skill | `db-decision-records` + `db-migrations` | 결정·변경·DBMS | `adr/*.md`, `migrations/*.sql` |
| 6. 품질관리 | Skill | `data-quality-management` | 산출물 전체 | `…-데이터품질부적합대장.xlsx` |
| 7. 데이터분석 | Skill | `data-analysis` | 확정 모델·산출물 | `…-데이터분석노트.md` |

> 3·5·6·7 스킬은 ECC 원본(`architecture-decision-records`·`database-migrations`·`quality-nonconformance`)을 공공DB로 현지화한 버전이다. ECC 원본은 방법론 참조용으로 보존되어 있다.

## 작업공간 (절대 경로)

```text
docs/04. db-deliverables/{요구사항번호}/
├── {요구사항번호}-db-modeling.md            # 1 데이터모델링
├── {요구사항번호}-표준검토결과.xlsx          # 2 표준화검토 (metadata-agent)
├── {요구사항번호}-구조검증결과.md            # 3 데이터구조검증
├── {요구사항번호}-{문서명}.xlsx              # 4 산출물 7종
├── {요구사항번호}-산출물관리대장.xlsx        # 4 최종검토 취합
├── adr/ADR-NNNN-{제목}.md                    # 5 변경관리(결정)
├── migrations/{순번}_{설명}.up/.down.sql     # 5 변경관리(스키마)
├── {요구사항번호}-데이터품질부적합대장.xlsx  # 6 품질관리(NCR)
└── {요구사항번호}-데이터분석노트.md          # 7 데이터분석
```

- `{요구사항번호}`는 입력 `{요구사항번호}-db-modeling.md`(또는 사용자 지정)에서 얻는다.
- **대상 DBMS**: 모델링.md의 `대상 DBMS` 또는 사용자 지정. 기본 **Oracle 11g**. 3·5단계 스킬에 일관 전달한다.
- 중간 산출물은 삭제하지 않는다(사후 검증·감사 추적용).

## 워크플로우

### Phase 0: 컨텍스트 확인 (후속 작업 지원)

`docs/04. db-deliverables/{요구사항번호}/` 존재 여부로 실행 모드를 결정한다.

- **미존재** → 초기 실행. Phase 1부터 전체 진행.
- **존재 + 부분 수정 요청** → 부분 재실행. 해당 단계 스킬만 재호출하고 그 산출물만 갱신, 하위 의존 단계(예: 1 수정 시 2~7)를 재검증한다.
- **존재 + 새 입력** → 새 실행. 기존 폴더를 `{요구사항번호}_{YYYYMMDD_HHMMSS}/`로 보관 이동한 뒤 Phase 1.

### Phase 1~7: 단계 위임

위 구성 표 순서대로 호출하며, 각 단계의 게이트·환류를 적용한다. **진입 시 7단계를 TodoWrite로 등록**하고, 각 단계 진입 시 `in_progress`·완료(산출물 확인) 시 `completed`로 갱신한다.

- **2 표준화검토**: metadata-agent는 공공표준 우선, 없으면 기관(KOMSA)표준 생성·등록. 결과 반영만. `확인 필요`는 1단계로 환류.
- **3 데이터구조검증**: `적합/보완/부적합` 판정. `보완`/`부적합`은 1·2단계 환류, 전 항목 `적합` 전까지 4단계 보류(게이트).
- **4 산출물관리**: **산출물 7종을 상시 전부 생성한다(누락·일부 생성 금지)**. `data-output-management`를 호출할 때 7종 전체를 명시적으로 지시한다. 입력에 일부 정보(예: 관계·DB메타)가 없으면 그 항목은 `보완 필요`로 표기하되 **문서 자체는 생성**한다. ERD는 Mermaid `erDiagram`. 2·3 결과를 `표준검토상태`에 반영하고, 마지막에 산출물관리대장으로 취합한다.
  1. 논리데이터모델다이어그램 — `{요구사항번호}-논리데이터모델다이어그램.xlsx`
  2. 물리데이터모델다이어그램 — `{요구사항번호}-물리데이터모델다이어그램.xlsx`
  3. 엔터티정의서 — `{요구사항번호}-엔터티정의서.xlsx`
  4. 애트리뷰트정의서 — `{요구사항번호}-애트리뷰트정의서.xlsx`
  5. 데이터베이스정의서 — `{요구사항번호}-데이터베이스정의서.xlsx`
  6. 테이블정의서 — `{요구사항번호}-테이블정의서.xlsx`
  7. 컬럼정의서 — `{요구사항번호}-컬럼정의서.xlsx`
  - (+ 표준검토결과·산출물관리대장) **생성 후 7종 파일 존재를 확인**하고, 누락분이 있으면 재생성한다(게이트).
- **5 변경관리**: 설계 결정→ADR, 스키마 변경→가역·무중단 마이그레이션(DBMS 분기). 산출물 `변경이력`과 연계.
- **6 품질관리**: NCR·RCA·CAPA. 부적합은 3·4단계 환류, 미결 시 완료 보고 금지(게이트).
- **7 데이터분석**: 프로파일링·규모·코드분포·개인정보·연계·표준화효과 + 운영 권고.

### Phase 8: 정리

1. **산출물 7종 전부 생성 여부를 확인한다**(논리/물리ERD·엔터티·애트리뷰트·DB·테이블·컬럼정의서). 누락분이 있으면 4단계를 재호출하여 보완한다 — **컬럼정의서만 생성된 상태로 종료하지 않는다**.
2. 산출물관리대장으로 전 산출물·검토상태를 취합 확인한다.
3. 미결(보완/부적합/NCR) 0건인지 확인한다. 남아 있으면 미결 명시.
4. **TodoList 전 항목이 `completed`인지 확인**한다(미완 항목이 남았으면 해당 단계 보류 사유를 보고에 명시).
5. 작업공간을 보존하고 사용자에게 결과 요약(생성 산출물·미결·다음 액션)을 보고한다.

                       Agent─ metadata-agent ─→ 표준검토결과.xlsx

                                      │ (보완/부적합→1·2 환류 / 게이트)
                  Skill─ data-output-management ─→ {문서명}.xlsx ×7 + 산출물관리대장

        Skill─ db-decision-records + db-migrations ─→ adr/ + migrations/

## 에러 핸들링

| 상황 | 전략 |
|---|---|
| 단계 스킬/에이전트 실패 | 1회 재호출. 재실패 시 해당 단계 산출물 부재를 명시하고 후속 단계 보류 |
| 입력 `…-db-modeling.md` 부재 | Phase 1 중단, 입력 위치 요청 (추측 생성 금지) |
| metadata-agent `확인 필요` 다수 | 2→1 환류 후 사용자 확인. 미해소 항목은 `확인 필요`로 산출물에 표기 |
| 구조검증/품질 게이트 미통과 | 4·7 진행 보류, 환류 대상 단계 재실행. 미결을 보고서에 명시 |
| **산출물 7종 중 일부만 생성** | 4단계를 재호출하여 누락 문서를 생성한다. 7종(+산출물관리대장)이 모두 존재해야 Phase 8 통과(게이트) |
| 대상 DBMS 불명확 | 기본 Oracle 11g로 진행하되 보고서에 가정 명시, 필요 시 사용자 확인 |
| 부분 재실행 시 하위 의존 | 수정 단계의 하위 단계(예: 1→2~7)를 재검증, 미재검증분 명시 |

## 테스트 시나리오

### 정상 흐름
1. 사용자가 `FR-019-db-modeling.md`(대상 DBMS: Oracle) 제공.
2. Phase 0: 작업공간 미존재 → 초기 실행.
3. 1~7 순차 위임, 3·6 게이트 통과.
4. Phase 8: 산출물관리대장에 8개 산출물 `적합`, 미결 0건.
5. 예상 결과: `docs/04. db-deliverables/FR-019/` 하위 산출물 생성·보고.

### 에러 흐름
1. 3단계 구조검증에서 외래키 도메인 불일치 `부적합` 발생.
2. 게이트 작동 → 4단계 보류, 2단계(표준검토)로 환류.
3. metadata-agent 재검토로 도메인 정정 → 3단계 재실행 `적합`.
4. 4~7 진행, Phase 8에서 "FK 도메인 정정(ADR 기록)" 명시 보고.

## 경계

- 표준을 직접 확정하지 않는다 (metadata-agent 위임).
- 공공메타데이터를 수정하지 않는다.
- DDL을 실 DB에 실행하지 않는다 (마이그레이션 정의·산출물 생성까지).
- 요구사항에 없는 구조를 임의 설계하지 않는다.
- 대상 DBMS를 임의 변경하지 않는다.
- 중간 산출물을 삭제하지 않는다 (감사 추적 보존).
