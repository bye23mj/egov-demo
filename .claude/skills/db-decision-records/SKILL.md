# db-decision-records (공공DB 설계결정 ADR)

> 범용 ADR 포맷·라이프사이클(Michael Nygard 경량 ADR, 구 ECC `architecture-decision-records`)을 **공공데이터베이스 설계 결정**에 흡수·현지화한 자족 스킬이다.

## 사용 시점

- da-agent 5단계(변경관리)에서 설계 결정을 기록할 때
- 표준/도메인/정규화/DBMS 선택 등 대안 중 하나를 택했을 때
- "왜 이 표준용어/도메인/구조인가"를 질문받을 때(기존 ADR 조회)

## 결정 탐지 신호

대화에서 다음이 보이면 ADR 기록을 제안한다(사용자 확인 없이 자동 생성 금지).
- 명시적: "이걸로 가자", "X 대신 Y를 쓰자", "ADR로 남겨줘", "결정 근거는…"
- 암시적: 표준/도메인/DBMS/정규화 대안을 비교해 결론에 도달, 식별자·반정규화 결정

## 기록 대상 결정 (공공DB)

- **표준 선택**: 공공표준 적용 vs 기관(KOMSA)표준 신규생성 사유
- **도메인 매핑**: 동일 형식단어에 복수 도메인 후보가 있을 때의 선택 근거
- **정규화 수준**: 반정규화 허용 여부와 사유(성능·이력관리 등)
- **식별자 설계**: 자연키 vs 인조키, 주식별자 선정
- **대상 DBMS**: Oracle vs PostgreSQL 선택 및 그에 따른 설계 영향
- **개인정보 처리**: 암호화 도메인 적용·분리 저장 결정

## ADR 포맷

```markdown
# ADR-NNNN: [결정 제목]

**날짜**: YYYY-MM-DD
**상태**: 제안 / 채택 / 폐기 / 대체(ADR-XXXX)
**결정자**: [관여자]

## 맥락
- 요구사항번호, 대상 엔터티/컬럼, DBMS
- 결정이 필요한 이유와 제약(2~5문장)

## 결정
- 무엇을 어떻게 하기로 했는가(1~3문장)

## 검토한 대안
### 대안 1: [이름]
- 장점 / 단점 / 기각 사유

## 결과
- 긍정 / 부정 / 위험
- 메타데이터·산출물·마이그레이션에 미치는 영향
```

## 작성 워크플로우

1. **초기화(최초 1회)**: 인덱스 파일이 없으면 사용자 확인 후 생성.
2. 결정의 핵심·맥락·대안·결과를 추출하고 번호를 부여(기존 ADR 스캔 후 증가).
3. 초안을 사용자에게 제시하고 **승인 후에만** 파일을 쓴다(거부 시 파기).
4. 인덱스에 1행(번호·제목·상태·날짜) 추가.

## 좋은 ADR 기준

- **구체적으로**: "주민등록번호C13 도메인 채택" (X "도메인 적용")
- **왜를 기록**: what보다 rationale이 중요
- **기각 대안 포함**, **결과를 정직하게**(모든 결정엔 trade-off), **2분 내 읽히게 짧게**
- 사소한 결정(명명·포맷)은 기록하지 않음. 대체된 ADR은 후속 ADR을 링크.

## 라이프사이클

```
제안 → 채택 → [폐기 | 대체(ADR-NNNN)]
```

## 저장 위치 및 인덱스

- 디렉토리: `docs/04. db-deliverables/{요구사항번호}/adr/ADR-NNNN-{제목}.md`
- 인덱스: 같은 폴더 `ADR-index.md`에 번호·제목·상태·날짜를 1행씩 누적

## 경계

- 결정의 표준 적합성 판정은 metadata-agent 결과를 근거로 인용한다(자체 판정 금지).
- 코드/데이터를 수정하지 않는다(결정 기록만).
