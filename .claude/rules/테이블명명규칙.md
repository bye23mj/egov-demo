# 공공데이터 테이블 이름 작성 규칙

## 1. 목적

이 문서는 공공데이터베이스 산출물 작성 및 DA Agent/Codex 자동화 시 사용할 수 있는 **공공데이터 테이블 이름 작성 규칙**을 정의한다.

테이블명은 단순한 DB Object명이 아니라, 논리 데이터 모델의 엔터티가 물리 데이터베이스에 구현된 결과이므로 **표준용어, 표준단어, 표준도메인, 기관표준, DB표준, 명명규칙**을 함께 고려하여 작성해야 한다.

---

## 2. 기본 원칙

| 구분 | 작성 규칙 |
|---|---|
| 한글 테이블명 | 논리DB에 부여할 테이블 한글 명칭으로 작성한다. |
| 영문 테이블명 | 실제 물리 DB에 생성할 테이블 이름으로 작성한다. |
| 기준 문서 | 표준용어정의서, 기관표준, DB표준, 데이터 설계 표준, 명명규칙을 따른다. |
| 관련 엔터티 | 엔터티정의서에 기록한 한글 엔터티명을 기준으로 작성한다. |
| 테이블 유형 | 일반, 파티션, 클러스터, 뷰, 코드, 마스터, 임시, 통계, 이력, 로그, 매핑 등으로 구분한다. |
| 표준 검토 | 공통표준, 기관표준, DB표준 준용 여부를 식별할 수 있어야 한다. |

---

## 3. 테이블명 구성 방식

테이블명은 다음 구조를 기본으로 한다.

```text
[업무영역] + [관리대상/엔터티] + [테이블유형]
```

예시:

| 목적 | 한글 테이블명 | 영문 테이블명 예시 |
|---|---|---|
| 사용자 기본정보 | 사용자기본 | TB_USER_BSC |
| 사용자 권한 매핑 | 사용자역할매핑 | TB_USER_ROLE_MAP |
| 메뉴 권한 관리 | 메뉴권한 | TB_MENU_AUTH |
| 공통코드 | 공통코드 | TB_CMM_CD |
| 공통코드 상세 | 공통코드상세 | TB_CMM_CD_DTL |
| 접속 이력 | 접속이력 | TB_CONN_HIST |
| 연계 송수신 로그 | 연계송수신로그 | TB_IF_SND_RCV_LOG |
| 통계 집계 | 사용자월별통계 | TB_USER_MM_STAT |
| 임시 적재 | 사용자임시 | TB_USER_TMP |

`TB_` 접두어는 기관 또는 사업의 DB Object 명명규칙에 따라 사용할 수 있다. 중요한 것은 프로젝트 전체에서 동일한 규칙을 일관되게 적용하는 것이다.

---

## 4. 한글 테이블명 작성 규칙

한글 테이블명은 업무담당자, DA, 개발자, 감리자, 메타데이터 담당자가 모두 이해할 수 있도록 업무 의미를 명확하게 표현해야 한다.

### 4.1 권장 규칙

```text
1. 업무 의미가 명확한 명사형으로 작성한다.
2. 엔터티정의서의 엔터티명과 최대한 일치시킨다.
3. 개발자 편의 명칭, 은어, 약칭, 임의 명칭을 사용하지 않는다.
4. 테이블 유형이 중요한 경우 이름에 반영한다.
5. 동일하거나 유사한 의미의 테이블명을 중복 생성하지 않는다.
6. 임시, 이력, 로그, 통계, 매핑, 코드 테이블은 유형을 명확히 드러낸다.
7. 한글 테이블명은 표준용어 또는 기관표준 용어와 의미상 충돌하지 않아야 한다.
```

### 4.2 좋은 예

```text
사용자
사용자역할
사용자접속이력
메뉴권한
공통코드
공통코드상세
연계송수신로그
민원신청
민원처리이력
배치실행이력
```

### 4.3 피해야 할 예

```text
사용자정보관리테이블
유저
회원자료
관리테이블1
임시데이터
테스트테이블
마스터
기본정보
기타정보
```

`기본정보`, `마스터`, `상세`, `임시데이터`, `기타정보`처럼 단독으로는 관리 대상이 불명확한 이름은 사용하지 않는다.

## 5. 영문 테이블명 작성 규칙

영문 테이블명은 실제 DB Object명이므로 DBMS 제약, 기관 명명규칙, 표준단어 영문약어명을 모두 고려해야 한다.

### 5.1 권장 규칙

```text
1. 대문자 영문과 언더스코어(_)를 사용한다.
2. 표준단어 영문약어명을 우선 사용한다.
3. 한글 테이블명과 의미가 1:1로 대응되게 작성한다.
4. 업무영역 또는 시스템 구분 접두어를 사용할 수 있다.
5. 테이블 유형 접미어를 일관되게 사용한다.
6. DB 예약어, 모호한 약어, 임의 축약어를 사용하지 않는다.
7. 지나치게 긴 이름은 표준약어를 사용하여 축약한다.
8. 동일 의미 테이블이 여러 영문명으로 생성되지 않도록 한다.
```

### 5.2 추천 패턴

```text
TB_[업무영역]_[관리대상]_[유형]
```

예시:

| 한글 테이블명 | 영문 테이블명 | 설명 |
|---|---|---|
| 사용자 | TB_USER | 사용자 기본 테이블 |
| 사용자역할 | TB_USER_ROLE | 사용자와 역할 관계 |
| 메뉴권한 | TB_MENU_AUTH | 메뉴 접근권한 |
| 공통코드 | TB_CMM_CD | 공통코드 |
| 공통코드상세 | TB_CMM_CD_DTL | 공통코드 상세 |
| 접속이력 | TB_CONN_HIST | 접속 이력 |
| 연계로그 | TB_IF_LOG | 인터페이스 연계 로그 |
| 배치실행이력 | TB_BAT_EXEC_HIST | 배치 실행 이력 |

## 6. 테이블 유형별 접미어 규칙

공공/SI 프로젝트에서는 테이블 유형이 이름에서 드러나야 산출물 관리, 영향도 분석, 메타데이터 관리가 쉬워진다.

| 유형 | 한글명 예시 | 영문 접미어 예시 | 설명 |
|---|---|---|---|
| 기본/원장 | 사용자, 부서, 민원신청 | 생략, `_BSC`, `_MST` | 기준 정보 또는 원장성 데이터 |
| 상세 | 민원신청상세 | `_DTL` | 기본 테이블의 상세 항목 |
| 이력 | 권한변경이력 | `_HIST` | 변경 이력 관리 |
| 로그 | 접속로그, 연계로그 | `_LOG` | 시스템 또는 업무 처리 로그 |
| 코드 | 공통코드 | `_CD` | 코드 그룹 또는 코드 항목 |
| 코드상세 | 공통코드상세 | `_CD_DTL` | 코드값 상세 |
| 매핑 | 사용자역할매핑 | `_MAP`, `_REL` | 다대다 관계 또는 연결 테이블 |
| 통계 | 월별접속통계 | `_STAT` | 통계성 데이터 |
| 집계 | 일별처리집계 | `_SUM`, `_AGGR` | 집계 결과 데이터 |
| 임시 | 사용자임시 | `_TMP` | 임시 적재 또는 처리용 |
| 백업 | 사용자백업 | `_BAK` | 백업용 데이터 |
| 인터페이스 송신 | 연계송신 | `_IF_SND` | 외부 연계 송신 데이터 |
| 인터페이스 수신 | 연계수신 | `_IF_RCV` | 외부 연계 수신 데이터 |

`MST`, `BSC`, `DTL`, `HIST`, `LOG`, `MAP`, `REL`, `STAT`, `TMP`, `BAK` 등은 기관 표준약어로 먼저 정의한 후 사용한다.

## 7. 엔터티와 테이블명의 관계 규칙

테이블명은 논리 모델의 엔터티명과 정합성을 유지해야 한다.

```text
엔터티 → 테이블
속성 → 컬럼
식별자 → PK / UK
관계 → FK
도메인 → 데이터 타입 / 길이
```

### 7.1 매핑 규칙

```text
1. 1개 엔터티가 1개 테이블로 구현되면 엔터티명과 테이블명을 최대한 일치시킨다.
2. 1개 엔터티가 여러 테이블로 분리되면 테이블 유형 또는 분리 기준을 이름에 반영한다.
3. 여러 엔터티가 하나의 테이블로 통합되면 관련 엔터티명을 모두 기록한다.
4. 관계 엔터티는 매핑, 관계, 연결 테이블로 명확히 표현한다.
5. 테이블정의서의 관련 엔터티명은 엔터티정의서의 한글 엔터티명을 사용한다.
```

예시:

| 논리 엔터티 | 물리 테이블 | 판단 |
|---|---|---|
| 사용자 | TB_USER | 정상 |
| 사용자권한 | TB_USER_AUTH | 정상 |
| 사용자, 권한 | TB_USER_AUTH_MAP | 여러 엔터티 관계 테이블 |
| 민원신청 | TB_PETN_APLY, TB_PETN_APLY_DTL | 기본/상세 분리 |
| 계약 | TB_CNTR_01, TB_CNTR_02 | 일련번호 가능하나 의미 있는 접미어 권장 |

## 8. 금지 또는 지양해야 할 테이블명

| 금지/지양 이름 | 문제점 |
|---|---|
| TB_DATA | 관리 대상이 불명확함 |
| TB_INFO | 정보의 대상이 불명확함 |
| TB_MASTER | 무엇의 마스터인지 알 수 없음 |
| TB_TEMP | 임시 대상이 불명확함 |
| TB_USER1, TB_USER2 | 분리 기준이 불명확함 |
| TB_NEW_USER | 시간이 지나면 신규 의미가 사라짐 |
| TB_OLD_USER | 백업, 이력, 폐기 구분이 불명확함 |
| TB_TEST | 운영 DB에 부적합함 |
| TB_ETC | 기타 테이블은 데이터 품질관리상 부적절함 |
| TB_TBL_USER | 테이블 의미가 중복된 접두어 사용 |

## 9. 테이블정의서 작성 시 필수 검토 항목

테이블 이름을 정의할 때 테이블정의서에는 최소한 다음 항목을 함께 관리한다.

| 항목 | 작성 기준 |
|---|---|
| 한글 테이블명 | 업무 의미가 명확한 한글명 |
| 영문 테이블명 | 물리 DB Object명 |
| 관련 엔터티명 | 엔터티정의서의 엔터티명 |
| 테이블 설명 | 목적, 포함 데이터, 제외 데이터, 업무규칙 |
| 테이블 유형 | 일반, 코드, 이력, 로그, 매핑, 통계, 임시 등 |
| 표준 적용 여부 | 공통표준, 기관표준, DB표준 준수 여부 |
| 개인정보 포함 여부 | 개인정보 또는 민감정보 포함 여부 |
| 보존기간 | 업무/법령 기준 보존기간 |
| 변경이력 | 생성, 변경, 폐기 이력 |

## 10. DA Agent / Codex 검증 체크리스트

```text
[테이블명 표준검토 체크리스트]

1. 한글 테이블명이 업무 의미를 명확히 표현하는가?
2. 한글 테이블명이 관련 엔터티명과 정합한가?
3. 영문 테이블명이 표준용어정의서 또는 표준단어 약어를 따르는가?
4. 영문 테이블명이 물리 DB Object 명명규칙을 준수하는가?
5. 테이블 유형이 이름 또는 테이블 유형 항목에 반영되었는가?
6. 이력/로그/임시/통계/코드/매핑 테이블의 유형이 명확한가?
7. 유사한 의미의 테이블명이 중복 생성되지 않았는가?
8. DB 예약어나 모호한 약어를 사용하지 않았는가?
9. 하나의 엔터티가 여러 테이블로 분리된 경우 분리 기준이 명확한가?
10. 여러 엔터티가 하나의 테이블로 통합된 경우 관련 엔터티명이 모두 기록되었는가?
11. DATA, INFO, MASTER, TEMP, ETC 등 단독으로 의미가 불명확한 명칭을 사용하지 않았는가?
12. TEST, OLD, NEW 등 시간이나 상태에 따라 의미가 변하는 명칭을 운영 테이블명에 사용하지 않았는가?
```

## 11. 적용 규칙

DA Agent가 테이블명을 생성할 때는 다음 규칙을 따른다.

```md
## 공공데이터 테이블 이름 작성 규칙

1. 테이블명은 공공데이터베이스 표준화 관리 기준에 따라 작성한다.
2. 한글 테이블명은 논리DB에 부여하는 테이블 한글 명칭으로 작성한다.
3. 영문 테이블명은 물리 DB에 생성할 테이블 이름으로 작성한다.
4. 한글/영문 테이블명은 표준용어정의서, 기관표준, DB표준, 명명규칙을 따른다.
5. 한글 테이블명은 관련 엔터티명과 의미상 일치해야 한다.
6. 영문 테이블명은 표준단어 영문약어명, 기관표준, DB표준 약어를 우선 사용한다.
7. 1개 엔터티가 여러 테이블로 분리되는 경우 유형명 또는 일련번호를 포함할 수 있다.
8. 테이블 유형은 일반, 파티션, 클러스터, 뷰, 코드, 마스터, 임시, 통계, 이력, 로그, 매핑 등으로 구분한다.
9. 테이블명에 테이블 유형이 반영된 경우 테이블 유형 항목은 생략할 수 있다.
10. 관련 엔터티명에는 엔터티정의서에 기록한 한글 엔터티명을 기재한다.
11. 여러 엔터티가 하나의 테이블로 통합되는 경우 관련 엔터티명을 모두 기재한다.
12. DATA, INFO, TEMP, MASTER, ETC 등 단독으로 의미가 불명확한 명칭은 금지한다.
13. TEST, OLD, NEW 등 시간이나 상태에 따라 의미가 변하는 명칭은 운영 테이블명에 사용하지 않는다.
14. 이력, 로그, 임시, 통계, 코드, 매핑 테이블은 이름에서 유형이 식별되도록 작성한다.
15. 표준에 없는 신규 테이블명은 DA Agent 또는 metadata-agent의 표준 검토를 거친 후 확정한다.
```

## 12. 최종 요약

공공데이터 테이블 이름 작성의 핵심은 다음 세 가지다.

```text
1. 표준용어정의서 기반 명명규칙 준수
2. 논리 엔터티와 물리 테이블 간 정합성 유지
3. 테이블 유형이 명확히 드러나는 이름 작성
```

즉, 테이블명은 개발자 편의가 아니라 **업무 의미, 표준 준수, 산출물 관리, 메타데이터 현행화, 변경관리**를 모두 고려하여 작성해야 한다.
