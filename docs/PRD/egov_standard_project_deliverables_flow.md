# 전자정부 표준프로젝트 기반 정부기관 프로젝트 표준 산출물 작성 흐름

## 1. 전체 산출물 작성 흐름 요약

정부기관 프로젝트의 산출물 흐름은 보통 아래 순서로 진행한다.

```text
사업착수
→ 현황분석
→ 요구사항 정의
→ 분석
→ 설계
→ 구현
→ 단위/통합 테스트
→ 시스템/인수 테스트
→ 보안·감리·품질 대응
→ 운영전환
→ 검수·종료
→ 하자보수
```

전자정부 표준프레임워크는 개발환경, 실행환경, 공통컴포넌트, 가이드 문서를 제공하는 기반이다. 실제 사업 산출물은 정보화사업 관리 지침, SW 개발 산출물 가이드, 감리 기준, 기관별 제안요청서에 맞춰 작성한다.

---

## 2. 산출물 작성의 핵심 원칙

정부기관 프로젝트 산출물의 핵심은 문서 개수보다 **추적성**이다.

```text
요구사항 ID
→ 유스케이스 ID
→ 화면 ID
→ API ID
→ 프로그램 ID
→ 테이블 ID
→ 테스트케이스 ID
→ 결함 ID
→ 검수 승인 ID
```

하나의 고객 요구사항이 어디에서 설계되고, 어떤 프로그램으로 구현되었고, 어떤 테스트로 검증되었으며, 어떤 검수 결과로 승인되었는지를 끝까지 연결해야 한다.

---

## 3. 단계별 표준 산출물 작성 흐름

### 3.1 사업착수 단계

#### 목적

사업 범위, 일정, 조직, 의사소통 방식, 품질관리, 보안관리, 산출물 관리 기준을 확정한다.

#### 주요 산출물

| 산출물 | 작성 주체 | 설명 |
|---|---|---|
| 착수계 | 사업자 | 사업 수행 의지와 착수 공식 문서 |
| 사업수행계획서 | PM/PL | 범위, 일정, 조직, 방법론, 산출물, 품질, 보안, 위험관리 포함 |
| WBS / 일정계획서 | PM | 단계별 일정, 마일스톤, 검토 일정 |
| 인력투입계획서 | PM | 투입 인력, 역할, 기간 |
| 산출물관리계획서 | PMO/PM | 산출물 목록, 작성자, 검토자, 승인자, 제출 시점 |
| 품질관리계획서 | 품질담당 | 리뷰, 테스트, 결함관리, 품질지표 |
| 위험관리계획서 | PM | 위험 식별, 대응계획 |
| 의사소통계획서 | PM | 정기회의, 보고체계, Slack/Jira/Confluence 사용 기준 |
| 보안관리계획서 | 보안담당 | 보안서약, 개발PC, 접근권한, 자료반출입, 개인정보 관리 |
| 형상관리계획서 | 개발리더 | Git 브랜치 전략, 커밋 규칙, 배포 태그 기준 |

#### 작성 흐름

```text
제안요청서/RFP 분석
→ 계약 범위 확인
→ 착수회의
→ 사업수행계획서 작성
→ 산출물 목록 확정
→ 기관 검토
→ 착수 승인
```

#### Jira / Confluence / Slack 적용

| 도구 | 활용 |
|---|---|
| Jira | 전체 WBS를 Epic/Task로 등록 |
| Confluence | 사업수행계획서, 산출물관리계획서, 회의체계 문서화 |
| Slack | 착수 공지, 회의 알림, 이슈 공유 |

### 3.2 현황분석 단계

#### 목적

기관의 현재 업무, 시스템, 데이터, 연계, 문제점, 개선 요구를 파악한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 현행업무분석서 | 현재 업무 프로세스, 담당자, 처리 절차 정리 |
| 현행시스템분석서 | 기존 시스템 구성, 메뉴, 기능, 인터페이스, 운영환경 분석 |
| 현행데이터분석서 | 기존 DB, 코드, 마스터, 트랜잭션 데이터 분석 |
| 인터페이스 현황표 | 대내외 연계 시스템, 방식, 주기, 데이터 항목 |
| 문제점 및 개선방향 | 현행 문제, 개선 필요사항, 기대효과 |
| 이해관계자 목록 | 업무담당자, 승인자, 운영자, 보안담당자 등 |
| 고객 인터뷰 회의록 | 인터뷰 내용, 결정사항, 액션아이템 |

#### 작성 흐름

```text
자료 요청
→ 기존 문서/시스템 분석
→ 담당자 인터뷰
→ 현행 업무 흐름 정리
→ 문제점 도출
→ 개선 방향 합의
→ 현황분석 산출물 승인
```

이 단계에서는 고객 인터뷰 회의록을 단순 회의록으로 끝내지 않고 요구사항 후보로 전환해야 한다.

### 3.3 요구사항 정의 단계

#### 목적

고객 요구사항을 공식화하고, 향후 설계·개발·테스트의 기준선을 만든다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 요구사항정의서 | 기능, 비기능, 보안, 성능, 연계, 데이터, 운영 요구사항 |
| 요구사항추적표 | 요구사항과 설계·개발·테스트 산출물 연결 |
| 요구사항명세서 | 각 요구사항의 상세 조건, 처리 규칙, 예외 |
| 요구사항 변경관리대장 | 범위 변경 요청, 영향도, 승인 여부 |
| 고객 인터뷰 회의록 | 요구사항 출처 근거 |
| 요구사항 승인서 | 고객 또는 주관기관 승인 기록 |

#### 작성 흐름

```text
회의록/인터뷰 내용 수집
→ 요구사항 후보 도출
→ 기능/비기능/보안/연계/데이터 요구사항 분류
→ 요구사항 ID 부여
→ 고객 검토
→ 수정
→ 기준선 확정
→ 요구사항 승인
```

#### 요구사항 ID 예시

```text
REQ-FN-001  민원 신청서 작성
REQ-FN-002  민원 신청서 임시저장
REQ-IF-001  행정정보공동이용 연계
REQ-SEC-001 개인정보 마스킹
REQ-PERF-001 목록 조회 3초 이내 응답
```

요구사항 승인 이후의 변경은 반드시 **Change Request**로 관리한다.

### 3.4 분석 단계

#### 목적

요구사항을 실제 시스템 기능 단위로 구조화한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 유스케이스 명세서 | Actor, 목적, 사전조건, 기본흐름, 대안흐름, 예외흐름 |
| 유스케이스 다이어그램 | 사용자와 시스템 기능 관계 |
| 업무흐름도 | To-Be 업무 프로세스 |
| 메뉴구조도 | 전체 메뉴 체계 |
| 화면목록 | 화면 ID, 화면명, 관련 요구사항 |
| 프로그램목록 | 프로그램 ID, 기능, 관련 화면/API |
| 인터페이스 요구사항 정의서 | 연계 대상, 방식, 데이터, 주기 |
| 데이터 요구사항 정의서 | 주요 데이터, 코드, 보존기간, 개인정보 여부 |

#### 작성 흐름

```text
요구사항정의서 기준 분석
→ Actor 식별
→ Use Case 도출
→ 업무흐름 정리
→ 화면/메뉴/프로그램 후보 도출
→ 요구사항추적표 갱신
→ 고객 검토
```

#### 핵심 연결 예시

```text
REQ-FN-001
→ UC-001 신청서 작성
→ SCR-001 신청서 작성 화면
→ API-001 신청서 저장 API
→ PGM-001 ApplicationController.save()
→ TC-001 신청서 저장 테스트
```

### 3.5 기본설계 단계

#### 목적

시스템 전체 구조, 아키텍처, 공통기능, 데이터 구조, 연계 구조를 설계한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 시스템아키텍처설계서 | 전체 구성, 서버, 네트워크, WAS, DB, 외부연계 |
| 애플리케이션아키텍처설계서 | Controller-Service-Repository/Mapper 계층 구조 |
| 공통기능설계서 | 로그인, 권한, 공통코드, 파일, 로그, 예외, 알림 |
| 데이터베이스설계서 | ERD, 테이블 목록, 컬럼 정의 |
| 인터페이스설계서 | API, 전문, 배치, 파일연계 설계 |
| 보안설계서 | 인증, 인가, 암호화, 접근통제, 개인정보 보호 |
| 화면설계서 초안 | 주요 화면 레이아웃, 화면 흐름 |
| API 설계서 | REST API URI, Method, Request/Response |
| 배치설계서 | 배치 목록, 주기, 처리 방식 |
| 표준프레임워크 적용설계서 | eGovFrame 적용 범위, 공통컴포넌트 활용 계획 |

#### 작성 흐름

```text
분석 산출물 기준 아키텍처 수립
→ 표준프레임워크 적용 범위 결정
→ 공통컴포넌트 사용 여부 결정
→ DB/화면/API/연계 기본설계
→ 보안·개인정보 설계 반영
→ 설계 검토회의
→ 기본설계 승인
```

전자정부 표준프레임워크 기반 프로젝트에서는 공통코드, 권한, 게시판, 파일, 로그인 등은 신규 구현 전에 공통컴포넌트 적용 가능성을 먼저 검토하는 것이 좋다.

### 3.6 상세설계 단계

#### 목적

개발자가 바로 구현할 수 있는 수준으로 화면, API, 클래스, SQL, 배치, 예외, 테스트 조건을 구체화한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 화면설계서 | 화면 항목, 버튼, 이벤트, 검증, 메시지 |
| 화면정의서 | 화면 ID, 권한, 입력/출력, 관련 API |
| 프로그램설계서 | 클래스, 메서드, 처리로직, 예외 |
| 컴포넌트설계서 | Controller, Service, Mapper, Validator 등 책임 |
| API 상세설계서 | Request/Response, 오류코드, 권한, 예시 |
| DB 상세설계서 | 테이블정의서, 컬럼정의서, 인덱스, 제약조건 |
| SQL/Mapper 설계서 | MyBatis Mapper, 주요 쿼리, 조건 |
| 권한설계서 | 역할별 메뉴·기능 접근권한 |
| 메시지/코드 정의서 | 공통코드, 오류코드, 안내 메시지 |
| 배치 상세설계서 | 배치 파라미터, 처리 순서, 실패 처리 |
| 테스트 시나리오 초안 | 기능별 정상/예외 테스트 조건 |

#### 작성 흐름

```text
기본설계 승인
→ 화면별 상세설계
→ API/프로그램/DB 상세설계
→ 권한/예외/메시지 설계
→ 개발자 리뷰
→ 고객 화면 리뷰
→ 상세설계 기준선 확정
```

#### 상세설계 작성 기준

```text
화면 ID
→ API ID
→ Service Method
→ Mapper ID
→ Table ID
→ Test Case ID
```

상세설계가 이 연결을 제공하지 못하면 개발·테스트 단계에서 추적이 끊긴다.

### 3.7 구현 단계

#### 목적

상세설계 기준으로 전자정부 표준프레임워크 기반 Java/Spring 코드를 구현한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 소스코드 | Controller, Service, Mapper, DTO, VO 등 |
| 소스코드목록 | 프로그램 ID, 클래스명, 경로, 설명 |
| 개발표준준수점검표 | 코딩 규칙, 네이밍, 예외, 로그, 보안 준수 여부 |
| 단위테스트케이스 | 클래스/메서드 단위 테스트 |
| 단위테스트결과서 | 테스트 수행 결과 |
| 코드리뷰결과서 | 리뷰 의견, 조치 결과 |
| 형상관리대장 | 브랜치, 커밋, 태그, 릴리즈 정보 |
| 빌드/배포 스크립트 | Maven/Gradle, CI/CD 스크립트 |
| 설정파일 목록 | application.yml, log 설정, profile 설정 |

#### 작성 흐름

```text
상세설계 확인
→ 개발 브랜치 생성
→ 코드 구현
→ 단위테스트 작성
→ 코드리뷰
→ 정적분석/보안점검
→ 개발서버 배포
→ 구현 산출물 갱신
```

#### 전자정부 표준프레임워크 적용 시 체크

| 항목 | 확인 내용 |
|---|---|
| 패키지 구조 | Controller / Service / Mapper / DTO / VO 분리 |
| 공통 예외 | 임의 예외 처리 대신 공통 예외 구조 사용 |
| 공통 응답 | API 응답 포맷 통일 |
| 공통코드 | 기관 코드/공통코드 체계 반영 |
| 로그 | 개인정보 로그 출력 금지 |
| 트랜잭션 | Service 계층에서 관리 |
| Mapper | SQL 책임 분리 |
| 보안 | 인증/인가, CSRF, XSS, SQL Injection 방어 |

### 3.8 테스트 단계

#### 목적

요구사항이 정상 구현되었는지 검증하고 결함을 추적한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 테스트계획서 | 테스트 범위, 일정, 환경, 담당자 |
| 단위테스트결과서 | 개발자 단위 테스트 결과 |
| 통합테스트시나리오 | 화면/API/연계 통합 흐름 |
| 통합테스트결과서 | 통합 테스트 수행 결과 |
| 시스템테스트시나리오 | 전체 업무 흐름 기준 테스트 |
| 성능테스트계획서/결과서 | 응답시간, 처리량, 동시사용자 |
| 보안테스트결과서 | 취약점 점검, 조치 결과 |
| 사용자수용테스트 UAT 계획서 | 고객 검수 테스트 계획 |
| UAT 결과서 | 고객 테스트 결과 |
| 결함관리대장 | 결함 ID, 심각도, 원인, 조치, 재검증 |
| 요구사항추적표 갱신본 | 요구사항별 테스트 통과 여부 |

#### 작성 흐름

```text
테스트계획 수립
→ 테스트케이스 작성
→ 테스트 데이터 준비
→ 단위테스트
→ 통합테스트
→ 시스템테스트
→ 보안/성능테스트
→ UAT
→ 결함 조치
→ 재테스트
→ 테스트 완료 승인
```

#### 테스트 추적 예시

| 요구사항 | 테스트케이스 | 결함 | 결과 |
|---|---|---|---|
| REQ-FN-001 신청서 작성 | TC-001 | - | Pass |
| REQ-FN-002 임시저장 | TC-002 | BUG-014 | 재테스트 Pass |
| REQ-SEC-001 개인정보 마스킹 | TC-031 | BUG-022 | 조치중 |

### 3.10 운영전환 단계

#### 목적

개발 완료 시스템을 운영기관이 안정적으로 인수하고 운영할 수 있도록 준비한다.

#### 주요 산출물

| 산출물 | 설명 |
|---|---|
| 운영전환계획서 | 전환 일정, 대상, 담당자, 절차 |
| 배포계획서 | 배포 순서, 배포 파일, 롤백 절차 |
| 데이터전환계획서 | 기존 데이터 이관 방식, 검증 방법 |
| 데이터전환결과서 | 이관 건수, 오류 건수, 검증 결과 |
| 운영자매뉴얼 | 시스템 운영 방법, 배치, 로그, 장애 대응 |
| 사용자매뉴얼 | 사용자 기능 설명 |
| 관리자매뉴얼 | 관리자 메뉴, 권한, 코드 관리 |
| 설치매뉴얼 | 서버 설치, 환경설정, 배포 절차 |
| 장애대응절차서 | 장애 접수, 분류, 조치, 보고 |
| 백업/복구계획서 | 백업 주기, 보관, 복구 절차 |
| 인수인계서 | 산출물, 계정, 서버, 소스, 라이선스 목록 |
| 교육자료 | 사용자/운영자 교육 자료 |

#### 작성 흐름

```text
운영환경 준비
→ 배포 리허설
→ 데이터전환 리허설
→ 운영자 교육
→ 사용자 교육
→ 운영매뉴얼 검토
→ 전환 승인
→ 운영 전환
```

최근 공공 정보시스템 운영에서는 예방점검체계와 표준운영절차, 즉 SOP의 중요성이 커지고 있다. 따라서 운영전환 단계에서 장애대응절차서, 예방점검표, 배치점검표, 백업복구절차서를 별도 산출물로 두는 것이 좋다.

## 7. Jira · Confluence 기준 관리 흐름

### 7.1 Jira

Jira는 **작업과 상태 관리** 중심으로 사용한다.

| Jira 이슈 유형 | 연결 산출물 |
|---|---|
| Epic | 업무영역, 기능군 |
| Requirement | 요구사항정의서 |
| Use Case | 유스케이스명세서 |
| Story | 화면/API/프로그램 개발 |
| Task | 산출물 작성, 환경구성, 회의 액션 |
| Bug | 결함관리대장 |
| Change Request | 요구사항 변경관리대장 |
| Test | 테스트케이스 |
| Release | 배포계획서, 릴리즈노트 |

### 7.2 Confluence

Confluence는 **공식 산출물 저장소**로 사용한다.

```text
00_사업관리
01_요구사항
02_분석
03_설계
04_개발
05_테스트
06_감리_보안_품질
07_운영전환
08_검수_종료
09_하자보수
```

### 7.3 Slack

Slack은 **논의와 알림**까지만 담당한다.

```text
Slack 논의
→ 결정사항은 Confluence 회의록에 등록
→ 작업은 Jira 이슈로 등록
→ 변경요청은 Jira Change Request로 등록
```

## 8. 산출물 승인 흐름

정부기관 프로젝트에서는 문서를 만들었다고 끝이 아니다. 아래 승인 흐름이 필요하다.

```text
작성
→ 내부 리뷰
→ PM/PL 검토
→ 품질담당 검토
→ 고객 검토
→ 보완
→ 승인
→ 기준선 등록
→ 변경관리 대상 전환
```

| 단계 | 승인 대상 | 승인자 |
|---|---|---|
| 착수 | 사업수행계획서, WBS, 산출물계획 | 발주기관 PM, 수행사 PM |
| 요구사항 | 요구사항정의서, 요구사항추적표 | 업무담당자, 발주기관 PM |
| 분석 | 유스케이스, 업무흐름, 메뉴구조 | 업무담당자 |
| 설계 | 화면/API/DB/아키텍처 설계 | 발주기관 PM, 기술담당 |
| 테스트 | 테스트계획, 테스트결과 | 품질담당, 업무담당자 |
| 운영전환 | 매뉴얼, 전환계획, 교육자료 | 운영부서 |
| 종료 | 완료보고서, 인수인계서 | 발주기관 검수자 |

## 10. 최종 권장 작성 흐름

정부기관 프로젝트에서는 아래 흐름을 표준으로 잡으면 된다.

```text
[착수]
사업수행계획서 작성
→ 산출물관리계획서 확정
→ WBS/Jira 등록

[요구사항]
회의록 작성
→ 요구사항정의서 작성
→ 요구사항 ID 부여
→ 요구사항추적표 생성
→ 고객 승인

[분석]
유스케이스 작성
→ 업무흐름도 작성
→ 메뉴/화면/프로그램 목록 작성

[설계]
아키텍처 설계
→ DB 설계
→ 화면 설계
→ API 설계
→ 컴포넌트 설계
→ 보안/권한 설계

[구현]
전자정부 표준프레임워크 기반 개발
→ 코드리뷰
→ 단위테스트
→ 형상관리

[테스트]
통합테스트
→ 시스템테스트
→ 보안/성능테스트
→ UAT
→ 결함 조치

[운영전환]
배포계획
→ 데이터전환
→ 매뉴얼 작성
→ 교육
→ 인수인계

[검수/종료]
산출물 취합
→ 최종 요구사항추적표 확정
→ 완료보고
→ 검수
→ 하자보수 전환
```

## 11. 실무적으로 가장 중요한 산출물 10개

| 2 | 요구사항정의서 | 범위 기준 |
| 3 | 요구사항추적표 | 감리·검수 핵심 |
| 4 | 회의록/결정사항관리대장 | 고객 합의 근거 |
| 5 | 화면설계서 | 고객 리뷰 핵심 |

| 7 | DB설계서 | 데이터 구조 기준 |

## 12. AI 협업 도구 적용 운영안

전자정부프레임워크 + Jira + Confluence + Slack + Claude/Gemini/Codex 조합이라면 아래처럼 운영하는 것이 적합하다.

```text
Claude
→ 회의록 요약, 요구사항 도출, 유스케이스/컴포넌트 설계 초안

Gemini / Codex
→ Java + Spring 코드 구현, 테스트 생성, 리팩토링, 코드리뷰 보조

Jira
→ 요구사항, 변경요청, 결함, 테스트, 일정 추적

Confluence
→ 공식 산출물 저장소, 승인 이력 관리

Slack
→ 회의 알림, 리뷰 요청, 이슈 공유
```

가장 중요한 규칙은 다음과 같다.

```text
Slack에서 논의된 내용은 Confluence 회의록으로 남긴다.
Confluence에서 확정된 요구사항은 Jira Requirement로 등록한다.
Jira Requirement는 설계서, 코드, 테스트케이스와 연결한다.
고객 변경은 기존 요구사항 수정이 아니라 Change Request로 분리한다.
완료 시 최종 요구사항추적표로 검수한다.
```

## 13. 결론

전자정부 표준프로젝트 기반 정부기관 사업의 산출물 작성 흐름은 단순히 문서를 많이 작성하는 것이 아니라, 다음 연결을 끝까지 유지하는 것이다.

```text
요구사항
→ 분석
→ 설계
→ 구현
→ 테스트
→ 보안·품질·감리
→ 운영전환
→ 검수
```

이 흐름에서 가장 중요한 관리 대상은 **요구사항정의서, 요구사항추적표, 회의록, 설계서, 테스트결과서, 결함관리대장, 운영전환 산출물**이다.

Jira는 상태와 작업을 관리하고, Confluence는 공식 산출물을 관리하며, Slack은 커뮤니케이션과 알림 용도로 제한하는 것이 안정적이다.
